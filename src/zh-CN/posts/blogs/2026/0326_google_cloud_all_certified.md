## Google Cloud 认证的契机
这件事的起因是我在2025年夏天参与的一个项目。虽然该项目在 Google Cloud 上进行开发，但我负责的是 AWS 端的 API 开发，实际上并没有接触 Google Cloud 环境。然而，我想着“今后也应该了解一下”，于是以学习为目的参加了 Google Cloud 的 “Associate Cloud Engineer (ACE)” 考试并通过了。

恰好那段时间赶上了 AWS 认证的更新期，所以我先优先完成了那部分。到了2025年12月，当我满足了“Japan AWS All Certifications Engineers”的条件后，正式开始全力冲刺 Google Cloud 认证全冠。

另外，Google Cloud 认证的考试供应商在2026年2月23日之前由 Kryterion 提供，但从3月开始改为 Pearson VUE。

在我所在地区，Kryterion 时代只能选择指定的测试中心；而改为 Pearson VUE 后，可以选择多个测试中心。此外，Kryterion 在考试日期更改上会在考试前72小时锁定（可付手续费更改），而 Pearson VUE 则可在24小时前更改，这对考生而言是极大的好消息。

顺便一提，这两个月里我去了同一个测试中心11次（其中2次同日考试），已经被工作人员记住了。有一次与他们闲聊时，听说他们正在准备支持 Pearson VUE，果然从3月中旬开始就已支持，这意味着我今后还能继续使用这个熟悉的测试中心。

### 注意考试费用以“美元结算”
关于测试中心的附带话题，还有一点是我在短期内大量考试时深切体会到的——**考试费用的结算货币**差异。

AWS 认证的考试费用以日元（JPY）结算，而 Google Cloud 认证的考试费用则在供应商页面上以**美元（USD）结算**。Professional 级别的考试一次需支付200美元，因此如果在短期内连续参加这么多考试，信用卡账单将直接受到汇率波动的影响。若使用公司的资格获取支持制度进行费用报销，建议在预算申请时预留足够的汇率变动空间。

## 整体难度：与 AWS 认证的比较
这么短时间内能通过多门考试，原因在于**“大部分通过的认证内容都可以靠对 AWS 知识的复用来应对”**。此外，我也深感充分理解 Google 提倡的 SRE（Site Reliability Engineering）十分重要。

另外，在文字量上也存在难度差异。AWS 认证的 Professional 和 Specialty 级别的题干和选项都较冗长，需要费力理解；而 Google Cloud 认证的 Professional 级别文字量几乎相当于 AWS 认证的 Associate 级别，阅读负担并不大。

### 掌握 Google Cloud 考试的关键：“SRE 基本概念”
在多门考试（尤其是 Cloud Architect、DevOps Engineer 等）中，Google 提出的**SRE（Site Reliability Engineering）**核心概念作为共通语言频繁出现。只要掌握这些，就能在情景题中迅速选择符合 Google Cloud 推荐的“正确行动”。

- **SLI / SLO / SLA 的区别**：SLI 是衡量服务可靠性的指标；SLO 是开发团队与运维团队的共同目标值；SLA 是与客户签订的业务合同，它们各自承担不同角色。
- **错误预算（Error Budget）**：不是追求 100% 可用性，而是设定“可容忍的故障预算”。在预算范围内优先发布新功能，预算耗尽后则优先提升可靠性（修复缺陷等）。
- **减少重复性劳动（Toil）**：将那些手工重复且无法自愈的运维工作（Toil），通过系统化或自动化尽可能地减少。
- **无责归因复盘（Blameless Postmortem）**：故障发生时，不责怪特定个人，而是分析系统或流程的缺陷，专注于为自家系统构建防范再发机制的文化。

## AWS 与 Google Cloud 概念差异（考试注意点）
虽说很多题目只要具备 AWS 知识就能解答，但在架构基本概念上也存在一些决定性差异，考试会对此进行考察。以下列举几项代表性的：

- **VPC 的作用域**：AWS 的 VPC 在特定“区域（Region）”内创建，而 Google Cloud 的 VPC 是“全球”资源。VPC 下创建的子网与各区域关联，因此跨区域网络构建的思路大相径庭。
- **资源管理单元（账户与项目）**：在 AWS 中，将“AWS 账户”作为环境和权限分离的边界；而在 Google Cloud 中，以“项目”为基本单位，通过“文件夹”或“组织”进行层级管理。
- **负载均衡器部署**：AWS 的主要负载均衡器（如 ALB）是区域资源；而 Google Cloud 的全球负载均衡（Cloud Load Balancing）使用单一 Anycast IP，将全球用户流量分发至最近区域。

## 各级别及特色考试的感想
Google Cloud 认证不像 AWS 那样有“高级资格通过自动更新低级资格”的机制。因此，需要在有效期（2年或3年）到期前单独更新各项认证。不过，对于临近到期的持证者，会提供不同于新考的**“更新用考试（Recertification Exam）”**，可按官方通知参加。

### Foundation 级别
50～60题。90分钟。99美元（税前）。3年有效。

Cloud Digital Leader (CDL) 和 Generative AI Leader (GAIL) 主要考察 Google Cloud 服务可用性及 AI 一般论等基础内容。虽然考试时间为 90 分钟，但大约 45 分钟就能完成。

### Associate 级别
50～60题。120分钟。125美元（税前）。3年有效。

Associate Cloud Engineer (ACE) 和 Associate Data Practitioner (ADP) 的内容可直接复用 AWS 知识，轻松应对。虽然有 120 分钟时限，我不到 1 小时就完成了。

另一个稍有不同的是 Associate Google Workspace Administrator (AGWA)。顾名思义，它不涉及 Google Cloud，而是以 Google Workspace 的管理（审计响应、离职与入职处理、企业合并等）为主。我当时管理着自有域名邮箱，又为使用 Gemini 等已迁移至 Google Workspace，所以对大部分概念都很熟悉，应付顺利。

### Professional 级别
50～60题。120分钟。200美元（税前）。2年有效。

如前所述，题干长度相当于 AWS 认证的 Associate 级别，阅读并不费力。但偶尔会出现长篇大论，务必警惕。

#### 特色题型
在 Professional Cloud Architect (PCA) 考试中，会在大约半个屏幕（可调节）显示案例公司说明，并基于该内容回答案例分析题。该考试内容预计将于 2026年3月30日更新，形式可能发生变化。

而 Professional Cloud Network Engineer (PCNE) 则满满出题达到了官方公布的上限“60题”。比其他 Professional 级别题目更多，对体力与集中力要求更高。我在答第39题时才注意到题量，对心理造成不小打击。建议考生一开始就确认题量。

## 唯一的壁垒：PSOE 的难度与反思
顺利推进全冠之路的途中，我在正值折返点的 Professional Security Operations Engineer (PSOE) 考试中失利，无败绩记录就此中断。

其他考试（AWS 认证或除 PSOE 外的 Google Cloud 认证）主要考察“如何设计”，而 PSOE 更强调“问题发生时如何应对”。所涉及的安全工具也较为特殊，我本身又不擅长英语，试图“通过做题理解英文”实在是错误策略。

PSOE 尚不支持日语。我曾期待随着向 Pearson VUE 迁移时能顺便本地化，但目前并无此迹象。尽管 Professional Cloud Database Engineer (PCDBE) 也是英语考试，但题目内容相对单一，总算能理解；而 PSOE 的复杂情境则让我苦战。

## 学习方法的转变与“Google Cloud Skills Boost”的利用
到目前为止，我一直利用第三方教材（如 Udemy 的练习题）进行备考。不过，由于部分练习教材已被删除，今后我将调整策略。

从现在开始，我计划充分利用官方学习平台“Google Cloud Skills Boost”。

### 什么是 Google Cloud Skills Boost？
这是 Google Cloud 官方提供的按需学习平台，具有以下特点：

- 实践型动手实验室：利用提供的临时 Google Cloud 环境，在真实控制台界面或 CLI 中动手操作学习。
- 针对认证考试的学习路径：根据各资格考试的出题范围，系统地整理了相应的课程和任务集。
- 日语概念解读：提供了丰富的视频和文档，解说各服务的概念和最佳实践。

对于像 PSOE 这样需要实际故障排除能力的考试，仅靠死记硬背是不够的，必须了解真实的行为。因此，我深刻体会到，应利用 Skills Boost 的动手实验，按**“先用日语理解概念和应对方法 → 再去阅读英文”**的顺序进行备考。

## 结语
约两个月无败全冠制霸的目标被唯一失利的 PSOE 阻挡，但也证明了，只要有 AWS 知识基础，跟上 Google Cloud 也非常顺利。

原本打算在三月中复仇，但因种种原因及教材重新评估，将其推迟到四月。四月我将通过 Skills Boost 采用新的学习方法向 PSOE 发起复仇，下次定能撰写“Google Cloud 全冠达成”的文章，加油！
